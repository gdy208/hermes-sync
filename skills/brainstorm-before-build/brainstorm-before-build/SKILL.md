# Brainstorm Before Build — Workflow projet

## Déclencheur
À chaque fois que l'utilisateur dit "j'ai un projet" ou "je dois rendre un projet" ou "on travaille sur un projet".

## Règle d'or
**Ne PAS coder/écrire avant validation du plan.** La phase de production ne commence qu'après un "go" explicite.

## Phase 1 — Brainstorm
1. L'utilisateur présente le sujet du projet
2. Poser des questions pour clarifier :
   - Objectif final ? (rendu, démo, production ?)
   - Contraintes ? (technos imposées, délais, ressources)
   - Quoi de déjà décidé VS ouvert ?
   - Critères de succès ?
3. Proposer 2-3 approches avec trade-offs
4. Laisser l'utilisateur réagir, débattre
5. Converger vers une direction

## Phase 2 — Validation du plan
1. Rédiger un plan d'attaque :
   - Architecture / structure
   - Étapes ordonnées
   - Technos utilisées
   - Livrables
2. **Ne pas passer à la production sans "go" explicite**

## Phase 3 — Production
1. Exécuter le plan étape par étape
2. Montrer l'avancement régulièrement
3. Ajuster si nécessaire

## Pitfalls
- Sauter la validation "c'est évident" — toujours valider
- Coder trop tôt une approche discutable
- Laisser l'utilisateur passif dans le brainstorm — le faire participer
